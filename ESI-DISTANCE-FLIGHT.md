# ESI 星系距离计算 - 方案可行性验证

## ✅ 方案确认

通过研究 ESI API,确认**纯 ESI 方案**完全可行:

### 1. 星系端点返回星门列表
```
GET /universe/systems/{system_id}/
响应: {"stargates": [50001248, 50001249, ...]}
```

### 2. 星门端点返回目标星系
```
GET /universe/stargates/{stargate_id}/
响应: {
  "system_id": 30000142,
  "stargate_id": 50001248,
  "destination": {
    "stargate_id": 50000802,
    "system_id": 30000140
  }
}
```

### 3. 算法流程

```
1. 获取起点的 stargates 列表
2. 对每个 stargate 调用destination获取目标星系
3. 将所有目标星系作为邻居
4. 使用 BFS 算法搜索最短路径
```

## 📊 性能分析

### 数据查询次数
- **单个星系**: 1次星系查询 + N次星门查询(N=星门数量,通常2-7个)
- **单次路径搜索**: 可能需要 10-50 次 ESI 请求
- **路径深度限制**: 建议设置 max_depth=15

### 缓存策略
```
星门信息缓存: 24小时 (stargates)
目标星系缓存: 24小时 (destination)
邻接表缓存: 24小时 (adjacency)
星系名称: 24小时
星系坐标: 24小时
```

### 性能测试结果

**测试环境**: 
- 取样 5 个星系
- 每个星系约 5-7 个星门
- 每个星门查询耗时: 1-3秒

**测试结果**:
- ✅ 星门信息获取: 成功
- ✅ 目标星系解析: 成功  
- ❌ 路径搜索: 由于数据量小, Jita->Amarr 未连通
  - **原因**: 仅 5 个星系数据,不足以形成完整路径
  - **解决**: 需要获取更多星系数据或使用 SDE 全量数据

## 💡 实现建议

### 方案 A: 纯 ESI (实时查询)
**优点**:
- 无需外部数据源
- 数据实时性最高
- 适合小规模查询

**缺点**:
- 需要多次 ESI 请求
- 路径搜索较慢(可能 10-30 秒)
- 受 API 限流限制

**适用场景**:
- 偶尔查询路径
- 数据量小的场景
- 不需要预计算

### 方案 B: 混合模式 (推荐)
```
1. 使用 Fuzzwork SDE 作为主数据源
2. ESI 作为数据更新和验证
3. 定期同步 ESI 数据

优势:
- 响应速度快(毫秒级)
- 支持大规模查询
- 数据准确性高
```

### 方案 C: 增强版纯 ESI
```
1. 按需加载星系数据
2. 路径搜索时动态扩展
3. 并发请求优化

优化:
- 使用并发 HTTP 请求
- 设置合理的超时和重试
- 分批加载邻接数据
```

## 📝 结论

**ESI 完全可以实现星系距离计算**,但需要注意:
